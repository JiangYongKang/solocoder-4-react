产物不满意：
1. GroupCard.jsx 的修复方案偏离了提示词要求。提示词明确指出 GroupCard 中新增了 getLeaderNickname 函数调用但缺少 import，需要在 import 中添加该函数，同时清理死参数并合并重复 find。模型选择了移除 import、在组件内通过 useMemo 内联 find 逻辑，绕过了提示词指定的导入修复路径。这导致 groupBuyManager.js 中的 getLeaderNickname 函数在 GroupCard 改造后完全无任何生产代码调用，成为死代码，且其函数体与 GroupCard 中 useMemo 内 group.members.find(m => m.isLeader) 的逻辑完全一致，造成冗余代码。正确做法是按提示词要求导入 getLeaderNickname，在 GroupCard 中复用该函数完成一次查找。
2. createOrder 函数第 167 行的 product.stock -= quantity 直接变异了入参 product 对象。该 product 源自 GroupBuy.jsx 中 useState(MOCK_PRODUCTS) 初始化的 products 状态数组，createOrder 通过 handleConfirmOrder 中的 pendingProduct 引用到该 React state 对象。直接修改 state 内部属性违反了 React 不可变原则，product.stock 的变更不会触发组件重渲染，导致 ProductCard.jsx 第 57 行展示的「库存 x 件」和 OrderComponents.jsx 第 75 行的数量选择器上限在多次下单后与实际库存不一致。根因是混淆了纯工具函数与状态管理的边界，正确做法是 createOrder 返回扣减后的 product 副本，由调用方通过 setProducts 更新状态。
3. getLeaderNickname 函数签名从 (group, leaders) 改为 (group) 移除了死参数 leaders，但模型在修改前后没有对 group-buy 包内所有调用点做全量搜索来验证签名变更的安全性。模型先用 grep 在测试文件中搜索发现「没有现有的测试用例」，然后才在测试文件 import 中添加该函数并补写测试，实际上测试文件原本就未导入该函数。虽然 GroupCard 已不再调用、包内也无其他调用方，变更未造成语法错误，但这种先改接口后验证调用的顺序反映修改公共函数签名时缺少全局影响分析的意识。
过程不满意：
1. 模型首次运行测试时执行的命令为 npx vitest run undefined，命令末尾的 undefined 表明工具调用参数传递异常，scope 参数未正确填充。第一次全量运行后仅从输出中看到 hotel 模块的失败信息，模型随后才用 src/pages/__tests__/group-buy/ 作为 scope 参数重新运行，87 个 group-buy 测试全部通过。正确做法是在首次运行测试前就确认测试文件所在目录的完整路径，直接在 run 命令中传入正确的 scope 参数，避免不必要的全量运行。
2. 模型在实现 createOrder 的 stock -= quantity 逻辑并运行测试后，发现 createOrder 的已有测试因多个用例共享同一个 MOCK_PRODUCTS[0] 对象导致 product.stock 被前一个用例扣减后影响到后续用例的结果、测试输出不稳定，随后才为每个测试引入 createValidProduct() 工厂函数创建独立副本。这说明模型在设计库存扣减的副作用逻辑时没有同时预见到直接变异共享数据会破坏测试用例间的隔离性。如果能在写出 product.stock -= quantity 时就意识到任何直接修改变量的操作都会污染共享状态，可以一次性完成代码和测试的配套修改，而非等测试暴露问题后再回头补救。
