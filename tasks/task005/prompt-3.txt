酒店预订页面中订单号生成的唯一性保证仍有缺陷：
1. generateOrderNo 函数虽然新增了自增计数器，但计数范围只有 0000-9999 四位数字，同一毫秒内生成超过 10000 个订单时计数器会归零，此后区分不同订单号完全依赖 4 位随机数后缀，碰撞概率不可忽略。这与"不依赖随机数偶然性"的要求相悖。需要彻底去除随机数后缀，使用足够位数的计数器（如 8 位或更多）或其他不依赖 Math.random 的方式保证订单号唯一。
2. validators.js 中新增的 resetOrderNoCounter 函数唯一调用方是测试文件的 beforeEach 钩子，该函数作为生产代码的公共 API 导出仅为了满足测试需要。需要将测试设施从生产代码的对外接口中移除，通过其他方式（如模块重加载或内部状态模拟）在测试中控制计数器状态。
3. 修复完成后重新运行 pnpm run test，需要让 hotel 相关的所有测试通过。
