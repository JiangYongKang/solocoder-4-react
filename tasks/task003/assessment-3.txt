产物不满意：
1. mapFilteredIndexToFullIndex 函数被直接定义在 KanbanPage 组件内部，未抽取为独立的可导出工具函数。该函数是本次修复的核心逻辑，包含五条决策分支，负责将筛选视图的任务索引映射到全量列表的真实位置。由于函数嵌在组件内且未导出，无法编写独立的单元测试来覆盖头部插入时 filteredIndex <= 0 分支、尾部插入时 filteredIndex >= filteredTasks.length 分支、中间插入时的锚点查找路径，以及锚点任务在 fullTasks 中找不到时的回退逻辑。根本原因是实现时只关注了功能本身的正确性，忽略了可测试性设计。正确做法是将该函数抽取到 utils.js 中并导出，为四个边界场景添加单元测试验证映射结果的准确性。
2. mapFilteredIndexToFullIndex 函数中 filteredTasks 为空数组且 filteredIndex 不为 null 的分支存在语义缺陷。当筛选列表为空时，KanbanColumn 不会渲染任何带 data-task-index 属性的 DOM 节点，因此 handleDragOver 中 taskWrapper 始终为 null，filteredIndex 只会保持为 null 并由第一个分支返回 fullTasks.length。该分支用 Math.min(Math.max(filteredIndex, 0), fullTasks.length) 做 clamp fallback，公式本身无法解释从空筛选列表中推导出有效全量索引的语义——若被未来的代码变更意外触发，会导致任务被插入到随机的 clamp 位置而非预期的末尾。正确做法是统一将该场景返回 fullTasks.length 表示追加到末尾，并注明该路径在当前数据流下不可达。
过程不满意：
3. 模型在完成 mapFilteredIndexToFullIndex 和 handleTaskDrop 的修改后直接运行了已有的 39 个测试用例并通过，但未为本次修复的核心映射逻辑新增任何单元测试。已有的测试全部针对 utils.js 中的纯工具函数，不涉及 KanbanPage 组件内部的拖拽索引映射。拖拽索引映射是整个修复中最容易出错的环节——涉及筛选视图与全量视图之间的坐标转换、同列拖拽的索引偏移修正、跨列拖拽的目标位置计算、以及空筛选列表和边界条件的处理，缺乏针对性的自动化测试意味着这五条决策分支的正确性完全依赖手工拖拽验证，回归风险未被控制。正确做法是在完成修复后至少为 mapFilteredIndexToFullIndex 编写筛选激活状态下向头部插入、向中间插入、向尾部插入和空筛选列表降级四种场景的单元测试，确保映射逻辑每个分支都有自动化验证。
